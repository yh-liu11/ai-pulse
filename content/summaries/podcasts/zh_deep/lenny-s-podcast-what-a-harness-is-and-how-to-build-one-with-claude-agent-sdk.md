## Summary

**简要总结**

AI harness（AI 套具）是一个将模型能力封装在结构化工作流、权限边界和工具适配器中的执行框架。Claire Vo 在节目中现场构建了一个基于 Claude Agent SDK 的 Sentry 错误调试套具，用于 ChatPRD 公司的生产环境。该套具自动完成证据收集、根因分析和后续工件创建，彻底消除了手动编写“请修复此 bug”提示词的需求。节目详细展示了套具的架构、代码结构以及构建过程，并给出了判断何时需要构建套具而非使用通用工具的具体标准。

**Core takeaways**

• **套具的定义**：套具不是模型本身，而是围绕模型构建的“脚手架”——它定义了模型能做什么、不能做什么、如何与外部系统交互，以及输出什么格式的工件。套具将一次性的对话式交互转变为可重复、可审计的结构化流程。

• **构建套具的时机**：当某个工作流具有高度重复性、需要严格的权限控制、涉及多个外部系统的协调，或者需要产出团队可共享的结构化输出时，应构建套具。对于探索性、一次性或高度可变的任务，通用工具（如 Claude Code 或 Codex）更为合适。

• **套具的三个必要组件**：每个套具必须包含（1）运行（runs）管理——追踪每次执行的完整生命周期；（2）任务（tasks）定义——将工作流分解为离散的、可验证的步骤；（3）工具（tools）和适配器——封装与外部系统（如 Sentry、Linear、GitHub、Vercel）的交互，并内置权限控制。

• **权限编码**：套具通过适配器层显式定义模型可以调用哪些 API、可以读取哪些数据、可以执行哪些写操作。这种设计将安全边界从提示词层面提升到代码层面，避免了提示词注入或模型越权操作的风险。

• **工件产出**：套具的输出不是对话文本，而是结构化的工件（如 Linear 工单、GitHub issue、根因分析报告），这些工件可以直接被团队其他成员或下游系统消费，无需人工二次处理。

**Details worth expanding**

• **架构设计**：套具采用分层架构。底层是 Claude Agent SDK，负责模型调用和基本的 agent 循环；中间层是自定义的任务编排器和工具适配器；顶层是基于 Ink 库构建的终端 UI，提供实时进度展示和交互控制。这种分层使得每一层可以独立测试和替换。

• **构建过程**：Claire 使用 GPT-5.5（Codex）和 Claude Opus 共同构建套具代码。她观察到两个模型在初始阶段都表现出“抗拒”——倾向于生成过于通用或过度简化的实现，而非针对 Sentry 错误调试这一特定场景进行深度定制。通过迭代式的提示词细化，最终获得了符合预期的代码结构。

• **代码布局**：套具的代码按功能模块组织，包括运行管理器、任务定义文件、工具适配器目录和 UI 组件。每个适配器（Sentry、Linear、GitHub、Vercel）都是独立的模块，暴露统一的接口，使得添加新的外部系统集成只需实现相同的接口契约。

• **现场演示结果**：节目中展示了套具处理一个真实 Sentry 错误的完整流程——从自动拉取错误详情和堆栈跟踪，到在 GitHub 中定位相关代码变更，再到在 Linear 中创建包含根因分析和修复建议的工单。整个过程无需人工干预。

**Implications for AI, investing, products, or research**

• **对 AI 产品开发的影响**：套具模式代表了 AI 应用从“对话界面”向“可编程工作流”的范式转变。未来的 AI 产品可能不再以聊天窗口为主要交互形式，而是以领域特定的套具为交付单元。这要求产品团队具备将领域知识编码为任务图和适配器的能力。

• **对投资判断的启示**：评估 AI 初创公司时，应关注其是否在构建领域套具而非仅仅包装通用模型。拥有深度行业适配器和结构化工作流的公司具有更高的护城河，因为其价值不仅来自模型能力，更来自对行业流程的编码和优化。

• **对研究方向的指引**：套具的权限编码和任务分解机制为 AI 安全研究提供了新的思路。通过在架构层面而非提示词层面实施约束，可以更可靠地控制 agent 行为。此外，套具的可组合性（多个套具协同工作）是一个值得探索的方向。

• **对工程实践的要求**：构建套具需要团队同时具备软件工程和领域专业知识。代码结构、错误处理、状态管理和可观测性等传统软件工程实践在套具开发中同样关键，甚至更为重要，因为 agent 的非确定性行为增加了调试难度。
